# Архитектура и данные

## Стек

- **Next.js 15** (App Router), **React 19**, **TypeScript**
- Сессия: **[iron-session](https://github.com/vvo/iron-session)** (зашифрованная cookie, имя `sb_session`)
- Пароли: хеш в `lib/password.ts` (не хранятся в открытом виде)
- Excel: библиотека **`xlsx`**
- Валидация: **`zod`** (точечно в API)

Продакшен **не требует** Chromium/Playwright на сервере: PDF формируется через **печать в браузере** пользователя.

## Маршруты приложения (UI)

После входа список проектов: `/app`. Проект: `/app/p/{id}/…`.

| Путь | Назначение |
|------|------------|
| `…/excel` | Загрузка и выбор документов `.xlsx`, шаблон |
| `…/events` | Редактор событий (черновик перед фиксацией сборки) |
| `…/timeline` | «Архитектура»: сетка по дням, стили, метки, раскладка |
| `…/tech-schedule` | Техрасписание |
| `…/rooms` | Аудитории и пересечения по расписанию |
| `…/responsibles` | Ответственные: фильтр по сотруднику, дни, проверка пересечений |
| `…/vks` | ВКС: статус по событиям + фильтр по дням |
| `…/broadcasts` | Трансляции: статус по событиям + фильтр по дням |
| `…/interpretation` | Синхронный перевод: статус по событиям + фильтр по дням |
| `…/volunteers` | Волонтеры: количество + фильтр по дням |
| `…/announcements` | Анонсы: конструктор текстового блока (мероприятие + шаблон + копирование) |
| `…/report` | Отчет: статистика по архитектуре/техрасписанию с учетом сокрытия |
| `…/export` | Обзор областей экспорта: Печать/PDF + Тильда для каждой вкладки |
| `…/export/print` | Страница печати (`pdfMode`, `pdfDay`, `printDialog`, `view`) |
| `…/export/tilda` | Экспорт HTML+CSS для T123 (в т.ч. `view`, вариант по дням) |

Авторизация: страница входа **`/sign-in`** (старый путь **`/login`** редиректится в `next.config.js`), API под `/api/auth/*`.

### Поток после "Создать проект"

- Форма на `/app` отправляет `POST` на `app/app/projects/route.ts`.
- В route handler создаётся проект (`createProject`) и выполняется `303` на `/app/p/{id}`.
- `app/app/p/[id]/page.tsx` делает направляющий редирект на `/app/p/{id}/excel` (первая рабочая вкладка проекта).
- Вкладки проекта отображаются в `app/app/p/[id]/layout.tsx` через `ProjectTabs`.
- В шапке проекта блок управления именем оформлен в одну строку из трёх элементов: поле названия, `Переименовать`, `Удалить проект`.

## HTTP API (route handlers)

Часть логики вынесена в `app/.../route.ts`:

- **`/api/auth/login`**, **`register`**, **`logout`** — вход и регистрация
- **`/api/auth/verify-email`** — подтверждение email по токену
- **`/api/auth/password/forgot`**, **`password/reset`** — восстановление пароля
- **`/app/projects`** — создание проекта (POST)
- **`/app/p/[id]/excel/upload`**, **`excel/template`** — загрузка и шаблон Excel
- **`/app/p/[id]/excel/import-url`** — импорт Google Sheets по ссылке (сервер скачивает `.xlsx`)
- **`/app/p/[id]/uploads`** — POST (form): удаление загрузки и файла на диске, редирект на `…/excel`
- **`/app/p/[id]/builds`**, **`builds/events`**, **`builds/timeline`**, **`builds/timeline-layout`**, **`builds/timeline-style`** — версии сборок и снимки событий/раскладки/стилей
- **`/app/p/[id]/export/tilda/data`** — JSON с полями `html` и `css`
- **`/app/p/[id]/export/tilda/snippet`** — `text/plain` сниппет для копирования в Тильду
- **`/api/export/tilda/data`**, **`/api/export/tilda/snippet`** — стабильные API-маршруты экспорта Тильда через query `projectId`

Все операции с проектом проверяют владельца через `getProject(projectId, userId)`.

## Поведение вкладок (May 2026 update)

- `rooms`, `volunteers`, `vks`, `broadcasts`, `interpretation`, `responsibles` используют единый паттерн периода: `Все дни` + кнопки конкретных дней программы.
- Дни строятся из реальных данных программы (`collectSortedProgramDayKeys*`), а не из фиксированного окна.
- Во вкладке `responsibles` добавлен выбор сотрудника из полного списка (поля `responsible1..6`), после чего список событий фильтруется по выбранному человеку.
- Конфликты ответственных считаются как **пары пересекающихся по времени событий** для выбранного человека в текущем периоде (`день` или `Все дни`).
- Во вкладке `События` обновлены UX-формулировки и действия добавления (`с указанием времени` / `без указания времени`), а панели редактирования имеют внутреннюю прокрутку для длинных форм.
- В панели `Редактирование` на вкладке `События` поля выстроены в единую вертикальную сетку с выровненными отступами; подписи `Редактирование` и служебные подзаголовки приведены к размеру `14px`.
- Вкладка `События`: формулировки чекбоксов видимости приведены к `Отображать`; в блоках переопределения стиля подпись `Цвет` выровнена по центру по высоте рядом с контролами.
- Оформление полей в редакторе событий унифицировано: `input/select/textarea` и `input[type=color]` используют единый радиус скругления.
- Для полей ввода в `Редактировании` добавлено явное правило полной ширины (`100%`) для всех не-чекбокс/не-range/не-color input, чтобы текстовые поля и поля даты не сужались относительно остальных.
- Счётчики количества событий в заголовках секций вкладки `События` выровнены по правому краю (вместо центрирования).
- Во вкладке `События` заголовки секций списка (в т.ч. `События без указанного времени` и заголовки дней) выравниваются по центру области.
- Поле `Сопроводительные материалы` в редакторе разделено на два независимых поля: `Сопроводительные материалы (Форматируемое)` и `Сопроводительные материалы (Загруженное)`.
- В навигации таймлайна для `Архитектуры` и `Техрасписания` используется единая терминология: `Период`, `Дней в периоде (От 1 до 10)`, `Показать скрытые дни`.
- Область настроек таймлайна разделена на независимые блоки `Сетка и расположение` и `Оформление` с отдельными кнопками сохранения; сверху закреплено пояснение, что именно сохраняет каждый блок.
- В блоке `Оформление` настройки выстроены построчно: для `Заголовка`, `Времени`, `Формата`, `Места` и для каждого допполя отдельные строки управления (`Тимлид`, `Ответственные`, `ВКС`, `Трансляция`, `Перевод`, `Волонтеры`); в каждой строке доступны размер, цвет, насыщенность и `Курсив`.
- В строках типографики таймлайна (`Оформление`) используется выравнивание по сетке во всю ширину блока: одинаковые контролы (`Размер`, `Цвет`, `Насыщенность`, `Курсив`) стоят в одинаковых колонках для всех строк.
- В строке `Название блока вечерней программы` лейбл и поле ввода расположены в одну строку (лейбл слева, input справа) с растяжением на всю ширину области `Оформление`.
- Внутри блока `Оформление` убраны лишние промежуточные подзаголовки (`Основные строки карточки`, `Дополнительные свойства событий`, `Цвета карточек и поля`) — оставлены только рабочие строки настроек.
- В таймлайне убрано служебное сообщение `Нет событий в этот день.` — при пустом дне блок сетки просто не отображается.
- Базовая типографика интерфейса унифицирована: заголовок страницы `16px`, подзаголовок `14px`, основной текст и кнопки `12px`.
- На таймлайне кнопка добавления пользовательской отметки времени подписана как `Добавить отметку`; в редакторе событий формулировка чекбокса приведена к `Показать в общем расписании`.
- В блоке `Метки времени` можно скрывать базовые отметки (построенные по стартам мероприятий), а также возвращать скрытые базовые отметки.
- Для отметок времени добавлены настройки внешнего вида: размер шрифта, цвет шрифта и цвет линии-разделителя.
- Описательные абзацы на страницах `Архитектура программы` и `Техническое расписание` отображаются на всю ширину рабочей области (без искусственного ограничения ширины).
- Во вкладке `Аудитории` добавлена проверка последовательности значений поля `Баннер` внутри выбранной аудитории:
  - учитываются только значения `Общий` и `Секционный` (`Не указано`/пусто игнорируются);
  - конфликт фиксируется при переключении `Общий <-> Секционный` у последовательных событий;
  - для дневного режима конфликт относится к дню первого события связки;
  - в режиме `Все дни` показываются все найденные конфликты за период.
- Во вкладках `ВКС`, `Трансляции`, `Перевод` отображаются только события со значением `Да`; значения `Нет`, `Не указано` и пустые значения не показываются.
- Во вкладке `Аудитории` добавлена проверка коротких перерывов: если пауза между соседними мероприятиями в одной аудитории меньше 20 минут, это выводится в отдельный блок проверки.

## Техрасписание: дополнительные поля

В режиме `showExtraFields` (`tech-schedule`) дополнительные поля событий показываются, если заполнены:

- `Тимлид` показывается отдельной строкой (только значение, без заголовка) над строкой `Ответственные`.
- Для флагов `ВКС`, `Трансляция`, `Перевод` показывается только название поля и только когда значение равно `Да`.
- Поля `Ответственные` и `Волонтеры` выводятся в явном виде при наличии данных.
- В шапке навигации техрасписания терминология периода: `Период` и `Дней в периоде (От 1 до 10)`.
- Для блока допсвойств (`Тимлид`, `Ответственные`, `ВКС`, `Трансляция`, `Перевод`, `Волонтеры`) настройки независимы между собой и не влияют друг на друга; значения сохраняются в `timeline_style`/`tech_timeline_style` активной сборки.
- На вкладке `Ответственные` добавлен блок проверки свободных ответственных на выбранную дату: отображается список сотрудников из общего перечня, у которых нет мероприятий в этот день.

## Новые поля исходного документа (May 2026)

В событие добавлены и поддерживаются на уровне импорта + редактора:

- `Тимлид` (`teamLead`) — строка (ФИО).
- `Фотографии от ответственного` (`photosFromResponsible`) — `Да | Нет | Не указано`.
- `Сопроводительные материалы` (`supportMaterials`) — текст.
- `Баннер` (`banner`) — `Общий | Секционный | Не указано`.

Эти поля:

- читаются из листа `Перечень` при парсинге;
- сохраняются в `events_json` сборки;
- доступны во вкладках `Анонсы` и `Отчет`;
- доступны для ручного редактирования в `…/events`.

## Вкладка "Анонсы" (конструктор текста)

- Пользователь выбирает мероприятие и один из шаблонов текста.
- На основе полей события формируется блок для копирования в соцсети/мессенджеры.
- В текст включаются: `Наименование`, `Описание`, `Формат`, `Дата`, `Время начала`, `Время окончания`, `Здание`, `Аудитория`.
- Для быстрого использования есть кнопка копирования в буфер обмена с fallback-механизмом, если `navigator.clipboard` недоступен.
- Добавлен блок редактируемых шаблонов: можно добавлять, изменять и удалять шаблоны прямо в UI (сохранение в `localStorage` браузера).
- В выпадающем списке мероприятий подпись содержит расширенные характеристики (дата/время/формат/площадка), чтобы различать события с одинаковым названием.
- Текст берётся из `ProjectEventIso`; в типе должны присутствовать как минимум `title`, `description`, `format`, `day/start/end`, `building`, `room`.

## Вкладка "Отчет" (статистика)

- Показатели считаются отдельно для `Архитектуры` и `Техрасписания`.
- В каждом блоке считаются:
  - общее количество мероприятий;
  - количество мероприятий с `Трансляция = Да`;
  - количество мероприятий с `ВКС = Да`;
  - количество мероприятий с `Синхронный перевод = Да`;
  - общее количество задействованных аудиторий (уникальные пары `Корпус + Аудитория`).
- Для расчета учитывается сокрытие в активной сборке:
  - `hidden_day_keys` (скрытые дни),
  - `event_overrides[dayKey][eventId].hidden` (скрытые карточки),
  - `visible=false` у события.

## Экспорт по областям

- На `…/export` доступны отдельные области экспорта: `Архитектура`, `Техрасписание`, `Аудитории`, `Ответственные`, `ВКС`, `Трансляции`, `Перевод`, `Волонтеры`.
- Для `print` и `tilda` передаётся query `view=<area>`.
- Для `view=tech-schedule` используются `tech_timeline_marks`, `tech_timeline_layout`, `tech_timeline_style`; для остальных областей — базовые `timeline_*`.

## Хранение данных

Единый файл состояния:

- **`data/store.json`** — пользователи, проекты, массив **сборок** (`builds`), ссылки на активный документ и активную сборку, кэш разобранного Excel в полях проекта.
- **`data/auth_tokens.json`** — служебные токены подтверждения email и сброса пароля (хранятся в виде hash, не в открытом виде).
- **`data/rate_limits.json`** — состояние rate-limit для auth-эндпоинтов.

Загруженные файлы:

- **`data/users/{userId}/projects/{projectId}/uploads/`** — сырые `.xlsx` и связанные имена в метаданных загрузки.

### Ограничения загрузок Excel

- Поддерживается только формат **`.xlsx`**.
- Лимит размера файла: `MAX_EXCEL_UPLOAD_BYTES` (по умолчанию 10 МБ).
- Лимит количества загрузок в одном проекте: `MAX_EXCEL_UPLOADS_PER_PROJECT` (по умолчанию 30).
- Нарушение ограничений возвращается как ошибка загрузки в UI (`?err=...` на вкладке `Документы`).
- Шаблон `Перечень` на вкладке `Документы` использует согласованный порядок колонок (включая сервисные поля `Рассадка`, `Вместимость`, `Количество участников`, `Контрагент или контактное лицо`, `Мебель`, `Комментарии`); лишние для текущей логики поля при импорте игнорируются.

### Импорт Google Sheets по ссылке

- На вкладке `Документы` доступен импорт через URL вида `https://docs.google.com/spreadsheets/d/<id>/edit?...`.
- Сервер строит экспортный URL `.../export?format=xlsx`, скачивает файл и прогоняет через тот же `parseXlsxBuffer`, что и обычный upload.
- Используется та же логика листа: только `Перечень` (или fallback на первый лист).
- Поддерживается только публичный доступ по ссылке (без OAuth).
- Дополнительные ошибки UI: `no_sheet_url`, `bad_sheet_url`, `sheet_not_public`, `sheet_fetch_failed`.
- На вкладке `Документы` отладочный блок с сырым `excel_json` в пользовательском UI не отображается.
- Область `Загрузка` оформлена как отдельный визуальный блок с двумя равнозначными зонами: `Загрузить файл` и `Импорт по ссылке`.
- Для полей `Выбрать файл` и `Ссылка Google Sheets` на вкладке `Документы` используется единый визуальный стиль (ширина, высота, скругление, отступы) для согласованного внешнего вида обеих зон загрузки.

### Где лежат пользовательские файлы и что можно чистить

- Все пользовательские загрузки: `data/users/**/projects/**/uploads/*`.
- Эти файлы можно удалять без вреда для системного кода, но при ручном удалении возможны "висячие" записи в `store.json`.
- Предпочтительный способ очистки: удаление через UI (вкладка `Документы`), чтобы одновременно чистились и диск, и метаданные.

Каталог **`data/`** создаётся при первом обращении к хранилищу (`lib/store.ts`, `lib/user-files.ts`). Его нужно **включать в бэкап** на продакшене и не затирать при деплое новой версии кода.

### Сборка (build)

- Создаётся из актуального набора событий (и привязки к исходной загрузке по необходимости).
- **Активная сборка** (`active_build_id`) — источник истины для экспорта (печать, Тильда) и для согласованности с «Архитектурой».
- В сборке хранятся, в частности: `events_json`, `timeline_layout` (высоты строк, колонки, скрытые дни и т.д.), `timeline_marks`, `timeline_style`.

## Безопасность (набросок)

- Доступ к проекту только у владельца (`user_id` в записи проекта).
- Секрет сессии задаётся переменной **`SB_SECRET`** (см. [deployment.md](deployment.md)).

Для публичного интернета дополнительно имеет смысл продумать rate limiting на `/api/auth/*`, HTTPS и заголовки безопасности — в базовом MVP это не зашито в код.

## Known regressions and fixes (May 2026 incident)

### Что сломалось

- На `/sign-in` возникала петля `ERR_TOO_MANY_REDIRECTS`.
- После части хотфиксов появлялись дубли `topbar`, "голый" интерфейс (неожиданная структура/стили), а на части маршрутов — белый экран.
- Регистрация падала на записи хранилища: `EACCES: permission denied, open '/app/data/store.json'`.

### Причины

- Redirect-логика была размещена на уровне layout, что дало конфликтующее поведение между публичной веткой и защищённой веткой.
- При изменениях layout нарушался единый контракт "кто рендерит каркас страницы", из-за чего UI мог рендериться дважды или без ожидаемой оболочки.
- В контейнере папка `/app/data` существовала, но у runtime-пользователя не всегда были права записи в том/каталог.

### Финальное состояние (зафиксировано)

- Root layout (`app/layout.tsx`) содержит только `<html>`, `<body>` и глобальные стили; бизнес-редиректы и auth-UI туда не помещаются.
- Protected layout (`app/app/layout.tsx`) отвечает только за общий контейнер защищённой зоны и не выполняет redirect.
- `topbar` рендерится ровно один раз в точках входа защищённого UI: `app/app/page.tsx` и `app/app/p/[id]/layout.tsx`.
- Проверки доступа выполняются в страницах и route handlers (`/app/*`, `/app/p/[id]/*`, `/app/projects`, `/api/...`).
- `/login` редиректится в `/sign-in` через `next.config.js` (одна точка правды для legacy URL).
- `app/sign-in/page.tsx` и `app/register/page.tsx` не используют `headers()`, чтобы избежать повторного 307/redirect-цикла в Next 15.
