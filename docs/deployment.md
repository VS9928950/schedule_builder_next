# Развёртывание (production)

## Сборка и запуск

На сервере (или в CI) из корня проекта:

```bash
npm ci
npm run build
NODE_ENV=production npm start
```

По умолчанию Next слушает порт **3000**. Порт можно задать так:

```bash
PORT=3000 NODE_ENV=production npm start
```

За приложением обычно ставят **обратный прокси** (nginx, Caddy, Traefik) с TLS и проксированием на `127.0.0.1:3000`.

## Переменные окружения

| Переменная | Обязательность | Назначение |
|------------|----------------|------------|
| `SB_SECRET` | **Обязательно в продакшене** | Секрет для шифрования cookie сессии (`iron-session`). Должен быть **длинным и случайным** (ориентир — не короче 32 байт в виде строки-секрета; см. требования iron-session к длине). |
| `NODE_ENV` | Рекомендуется `production` | Режим Next.js |
| `PORT` | По желанию | Порт HTTP-сервера `next start` |

В `lib/session.ts` для cookie в коде задано `secure: false`, что удобно для локального HTTP. На продакшене за **HTTPS** прокси это часто всё равно приемлемо для первых шагов; если нужна cookie только по HTTPS, логику `cookieOptions.secure` стоит сделать зависимой от окружения (отдельная правка кода).

## Данные и бэкапы

Сохраняйте и резервируйте каталог **`data/`** целиком:

- `data/store.json` — все пользователи и проекты;
- `data/users/**` — загруженные Excel и прочие файлы проектов.

При обновлении версии приложения **не удаляйте** `data/`, если не переносите данные осознанно.

## Обновление после выкладки

1. Остановить процесс `next start`.
2. Подтянуть новый код (`git pull` или артефакт CI).
3. `npm ci` / `npm install` при изменении `package-lock.json`.
4. `npm run build`.
5. Запустить `npm start` снова.

Пользователям не нужно ничего устанавливать: они получают новую версию при следующем открытии сайта.

## Мощность сервера (ориентир)

Для небольшой команды или умеренного числа одновременных пользователей обычно достаточно **1–2 vCPU**, **2 ГБ RAM**, **20+ ГБ** диска под ОС, приложение и рост `data/`. Подробнее — в обсуждении нагрузки и SQLite как единственной точки записи в `store.json` (при сильном росте конкурентных записей может понадобиться очередь или переход на клиент-серверную БД).

## Процесс под systemd / PM2

Пример с **PM2** (один инстанс):

```bash
npm run build
pm2 start npm --name schedule-builder -- start
pm2 save
```

Логи смотреть через `pm2 logs schedule-builder`. Конкретные unit-файлы systemd лучше составить под вашу ОС и пользователя сервиса.
