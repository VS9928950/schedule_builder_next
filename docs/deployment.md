# Развёртывание (production)

## Сборка и запуск

На сервере (или в CI) из корня проекта:

```bash
npm ci
npm run build
NODE_ENV=production npm start
```

По умолчанию Next слушает порт **3000**. Порт можно задать так:

```bash
PORT=3000 NODE_ENV=production npm start
```

За приложением обычно ставят **обратный прокси** (nginx, Caddy, Traefik) с TLS и проксированием на `127.0.0.1:3000`.

## Переменные окружения

| Переменная | Обязательность | Назначение |
|------------|----------------|------------|
| `SB_SECRET` | **Обязательно** | Секрет для шифрования cookie сессии (`iron-session`). Должен быть **длинным и случайным** (минимум 32 символа). Без переменной приложение не стартует. |
| `AUTH_EMAIL_REQUIRED` | По желанию | `true` — включить обязательное подтверждение email перед входом; `false` (по умолчанию) — отключить обязательную верификацию |
| `APP_BASE_URL` | **Обязательно в production** | Базовый URL приложения (например, `https://example.com`) для ссылок подтверждения email, сброса пароля и приглашений |
| `RESEND_API_KEY` | Рекомендуется для auth email | API ключ Resend (вариант по умолчанию для писем подтверждения и сброса) |
| `MAIL_FROM` | Рекомендуется для auth email | Адрес отправителя для email-сервисов |
| `YANDEX_SMTP_USER` | Опционально (если почта через Яндекс) | Логин SMTP Яндекс 360 (обычно полный email) |
| `YANDEX_SMTP_APP_PASSWORD` | Опционально (если почта через Яндекс) | Пароль приложения для SMTP (не основной пароль аккаунта) |
| `YANDEX_SMTP_HOST` | Опционально | SMTP host (по умолчанию `smtp.yandex.com`) |
| `YANDEX_SMTP_PORT` | Опционально | SMTP port (по умолчанию `465`) |
| `MAIL_WEBHOOK_URL` | Альтернатива | Внешний webhook отправки писем (если не используете Resend) |
| `MAIL_WEBHOOK_TOKEN` | По необходимости | Bearer token для `MAIL_WEBHOOK_URL` |
| `NEXT_PUBLIC_YANDEX_SMARTCAPTCHA_SITE_KEY` | Опционально | Публичный ключ виджета Яндекс SmartCaptcha (frontend) |
| `YANDEX_SMARTCAPTCHA_SERVER_KEY` | Опционально | Серверный ключ проверки Яндекс SmartCaptcha (backend) |
| `MAX_EXCEL_UPLOAD_BYTES` | По желанию | Лимит размера `.xlsx` (по умолчанию 10485760) |
| `MAX_EXCEL_UPLOADS_PER_PROJECT` | По желанию | Лимит количества `.xlsx` в одном проекте (по умолчанию 30) |
| `NODE_ENV` | Рекомендуется `production` | Режим Next.js |
| `PORT` | По желанию | Порт HTTP-сервера `next start` |

В `lib/session.ts` cookie `secure` включается автоматически при `NODE_ENV=production`. Для корректной работы на проде приложение должно стоять за HTTPS (reverse proxy с TLS).

### SMTP Яндекс 360 для писем регистрации/подтверждения/приглашений

- Включите пароли приложений в Яндекс-аккаунте.
- Создайте пароль приложения для почты.
- Задайте `YANDEX_SMTP_USER` и `YANDEX_SMTP_APP_PASSWORD` (при необходимости также `YANDEX_SMTP_HOST/PORT`).
- `MAIL_FROM` можно оставить пустым — тогда отправитель берётся из `YANDEX_SMTP_USER`.
- Эти настройки также используются для кнопки «Отправить приглашение на регистрацию» в админке.

### Яндекс SmartCaptcha для входа и регистрации

- Задайте `NEXT_PUBLIC_YANDEX_SMARTCAPTCHA_SITE_KEY` и `YANDEX_SMARTCAPTCHA_SERVER_KEY`.
- Если `YANDEX_SMARTCAPTCHA_SERVER_KEY` не задан, капча в API не требуется (мягкое отключение).

## Данные и бэкапы

Сохраняйте и резервируйте каталог **`data/`** целиком:

- `data/store.json` — все пользователи и проекты;
- `data/users/**` — загруженные Excel и прочие файлы проектов.
- `data/auth_tokens.json` — служебные токены email-верификации/сброса пароля.
- `data/rate_limits.json` — служебные counters rate-limit.

### Что можно удалять без поломки кода

- `data/users/**/projects/**/uploads/*` — загруженные пользователями Excel (потеряются сами файлы).
- `data/auth_tokens.json` — сервис восстановит файл автоматически (сбросятся активные ссылки подтверждения/сброса).
- `data/rate_limits.json` — сервис восстановит файл автоматически (сбросятся временные лимиты попыток).

Не удаляйте `data/store.json`, если нужно сохранить пользователей/проекты/метаданные.

При обновлении версии приложения **не удаляйте** `data/`, если не переносите данные осознанно.

## Обновление после выкладки

1. Остановить процесс `next start`.
2. Подтянуть новый код (`git pull` или артефакт CI).
3. `npm ci` / `npm install` при изменении `package-lock.json`.
4. `npm run build`.
5. Запустить `npm start` снова.

Пользователям не нужно ничего устанавливать: они получают новую версию при следующем открытии сайта.

## Known regressions and fixes (May 2026 incident)

Инцидент включал петлю редиректов на `/sign-in`, проблемы каркаса UI (дубли `topbar`/пустые экраны) и ошибку регистрации `EACCES` при записи `data/store.json` в контейнере.

Финально зафиксировано:

- `/login` всегда редиректится на `/sign-in` через `next.config.js`.
- Root/protected layout разведены по ролям (детали в `architecture.md`): root без auth-логики, protected без route-специфичной шапки.
- `Topbar` рендерится ровно один раз в `app/app/page.tsx` и `app/app/p/[id]/layout.tsx`.
- Для Docker в runtime-слое создаётся `/app/data` и выставляется владелец `nextjs:nodejs`, чтобы запись в `store.json` была доступна приложению.

## Deployment discipline (mandatory)

- Рабочая версия определяется только состоянием репозитория и GitHub-историей изменений.
- Процесс выпуска: изменить файлы в репозитории -> commit/push -> пересборка -> перезапуск контейнера.
- В браузере не выполняются "ремонты" приложения: браузер используется только для проверки результата уже задеплоенного кода.
- При любом споре "код vs наблюдение" первично проверяется корректность собственной диагностики относительно текущего кода и зафиксированных контрактов (`architecture.md`, `development.md`).

## Docker data permissions

Почему возникал `EACCES`:

- Приложение пишет пользователей/проекты в `process.cwd()/data/store.json` (`lib/store.ts`).
- В контейнере рантайм работает под непривилегированным пользователем `nextjs`.
- Если `/app/data` (или примонтированный volume) принадлежит другому пользователю/группе без записи, регистрация/создание проекта падают на `fs.writeFileSync`.

Как предотвращено:

- В `Dockerfile` добавлено создание каталога и назначение владельца до перехода на `USER nextjs`:
  - `RUN mkdir -p /app/data && chown -R nextjs:nodejs /app/data`
- В `docker-compose.yml` используется именованный том `app_data:/app/data` для постоянства данных между перезапусками.

Как проверить в контейнере:

```bash
docker compose exec app sh -lc "id && ls -ld /app /app/data && ls -l /app/data"
docker compose exec app sh -lc "test -w /app/data && echo writable || echo not-writable"
```

Ожидаемо: пользователь контейнера `nextjs` и запись в `/app/data` доступна.

## Post-deploy smoke checks

### 1) HTTP/curl

```bash
# Главная должна отвечать (200/307 допустимо в зависимости от сессии)
curl -I http://localhost:3000/

# Legacy login URL обязан вести на /sign-in
curl -I http://localhost:3000/login

# Публичные auth-страницы доступны без сессии
curl -I http://localhost:3000/sign-in
curl -I http://localhost:3000/register
```

Проверить, что для `/login` есть `Location: /sign-in`.

### 2) UI checks (браузер)

- На `/sign-in` и `/register` нет redirect loop, страница рендерится стабильно.
- На маршрутах `/app/*` верхняя панель (`topbar`) отображается ровно один раз.
- На `/app/p/{id}/*` видны вкладки проекта (`Документы`, `События`, `Архитектура`, `Техрасписание`, `Аудитории`, `Ответственные`, `ВКС`, `Трансляции`, `Перевод`, `Волонтеры`, `Экспорт`).
- Интерфейс не "голый": применены стили из `globals.css`, карточки/чипы/сетка выглядят штатно.
- Нет белого экрана при переходах между вкладками проекта.
- Во вкладках `Аудитории`, `ВКС`, `Трансляции`, `Перевод`, `Волонтеры`, `Ответственные` доступны кнопки дней + `Все дни`.
- Во вкладке `Ответственные` проверка пересечений показывает пары конфликтующих событий для выбранного сотрудника.

### 2.1) Export views checks

- На `/app/p/{id}/export` есть отдельные карточки областей экспорта:
  - `Архитектура`, `Техрасписание`, `Аудитории`, `Ответственные`, `ВКС`, `Трансляции`, `Перевод`, `Волонтеры`.
- Переход на `Печать / PDF` каждой карточки открывает `/export/print?view=<area>`.
- Переход на `Тильда` каждой карточки открывает `/export/tilda?view=<area>`.
- Для `view=tech-schedule` визуально используется техническая раскладка/стили (`tech_timeline_*`).

### 3) Registration/login checks

- Регистрация нового пользователя с валидным email/паролем завершаетcя переходом в `/app`.
- Невалидная регистрация показывает текст ошибки (не немой fail).
- Вход существующим пользователем ведёт в `/app`.
- Создание проекта на `/app` переводит в `/app/p/{id}/excel`.

## Мощность сервера (ориентир)

Для небольшой команды или умеренного числа одновременных пользователей обычно достаточно **1–2 vCPU**, **2 ГБ RAM**, **20+ ГБ** диска под ОС, приложение и рост `data/`. Подробнее — в обсуждении нагрузки и SQLite как единственной точки записи в `store.json` (при сильном росте конкурентных записей может понадобиться очередь или переход на клиент-серверную БД).

## Процесс под systemd / PM2

Пример с **PM2** (один инстанс):

```bash
npm run build
pm2 start npm --name schedule-builder -- start
pm2 save
```

Логи смотреть через `pm2 logs schedule-builder`. Конкретные unit-файлы systemd лучше составить под вашу ОС и пользователя сервиса.
