# Разработка

## Вход в код

- **Страницы и layout**: `app/`, вложенность повторяет URL.
- **Проект и вкладки**: `app/app/p/[id]/layout.tsx`, `ProjectTabs.tsx`, отдельные `page.tsx` под каждую вкладку.
- **Состояние и API**: `lib/store.ts` + соответствующие `route.ts` рядом с фичами.
- **Таймлайн**: `timeline/TimelineViewer.tsx` (большой компонент: сетка, drag, стили).
- **Печать**: `export/print/PrintWorkspaceClient.tsx`, стили печати в `app/globals.css`, масштаб в `lib/print-timeline-scale.ts`.

## Проверка типов

```bash
npx tsc --noEmit
```

## Локальные данные

При разработке создаётся **`data/`** в корне репозитория. Его можно удалить для «чистого» стенда (потеряются пользователи и проекты). В git каталог обычно не коммитят — при необходимости добавьте `data/` в `.gitignore`.

## Сессия в dev

`SB_SECRET` обязателен для любого окружения; без него приложение не стартует. Для локальной разработки задайте `SB_SECRET` (минимум 32 символа) в `.env.local`.

## Добавление фич

- Новые серверные эндпоинты — как **`export const dynamic`** не требуется по умолчанию; следите за вызовами `cookies()` и сессией только на сервере.
- Клиентские эффекты печати держите в компонентах с `"use client"`, чтобы не смешивать с RSC без нужды.
- Для вкладок отчётов (`rooms`, `volunteers`, `vks`, `broadcasts`, `interpretation`, `responsibles`) используйте единый day-filter паттерн: `Все дни` + список фактических дней программы.
- Для проверок пересечений явно документируйте метрику: число конфликтов = число пар пересекающихся событий.
- После каждого изменения кода обновляйте документацию в `docs/` в том же рабочем цикле (минимум `architecture.md` или профильный файл по фиче).

## Layout contract

### Root layout (`app/layout.tsx`)

- Разрешено: подключение `globals.css`, базовый `<html>/<body>`, общие метаданные.
- Нельзя: auth-redirect, проверка сессии, отрисовка protected-шапки или проектных вкладок.
- Причина: root применяется ко всем веткам (`/sign-in`, `/register`, `/app/*`), любая auth-логика здесь легко создаёт redirect loop или конфликт UI.

### Protected layout (`app/app/layout.tsx`)

- Разрешено: только единый каркас защищённой зоны (`wrap`), без route-specific UI.
- Нельзя: выполнять redirect на `/sign-in` или другие маршруты.
- Нельзя: рендерить `topbar` в этом layout (во избежание повтора при вложенных layout-цепочках).
- Redirect и проверка доступа должны оставаться в конкретных страницах `/app/*` и route handlers, чтобы не ломать публичные страницы и вложенные layout.

### Project layout (`app/app/p/[id]/layout.tsx`)

- Разрешено: загрузить проект, показать `topbar`, заголовок проекта и `ProjectTabs`.
- Redirect допустим только для валидации контекста проекта (`id` невалиден/проект не найден -> `/app`) и отсутствия сессии.
- Вкладки проекта должны рендериться именно здесь, один раз на страницу.

### Dashboard page (`app/app/page.tsx`)

- Рендерит `topbar` для маршрута `/app` (список проектов).
- Не должен дублировать route-специфичные элементы из project layout.

## Auth/public routes contract

- Публичные маршруты: `/sign-in`, `/register`.
- Дополнительно публичные auth-маршруты: `/verify-email`, `/forgot-password`, `/reset-password`.
- Защищённые маршруты: весь префикс `/app/*`.
- Legacy-маршрут `/login` не имеет собственной страницы и редиректит на `/sign-in` в `next.config.js`.
- Роли пользователей: `admin` и `user`.
- `admin` имеет доступ к `/app/admin` (управление пользователями); `user` в этот раздел не допускается и перенаправляется на `/app`.
- Для новых хранилищ первый зарегистрированный пользователь получает роль `admin`; при миграции legacy-хранилища без админа роль `admin` автоматически назначается первому пользователю.
- При наличии `YANDEX_SMARTCAPTCHA_SERVER_KEY` API входа и регистрации требуют валидный токен SmartCaptcha.
- Виджет SmartCaptcha на формах входа/регистрации включается через `NEXT_PUBLIC_YANDEX_SMARTCAPTCHA_SITE_KEY`.
- При настроенном SMTP в разделе `Администрирование` доступна отправка приглашения на регистрацию (email со ссылкой `/register`).
- Минимальная длина пароля унифицирована: 8 символов (регистрация, сброс, админ-операции).
- Для admin-операций (`create/password/delete/invite`) действует server-side rate-limit.
- При `AUTH_EMAIL_REQUIRED=true` после регистрации требуется подтверждение email; вход разрешён только для `email_verified`.
- При `AUTH_EMAIL_REQUIRED=false` (по умолчанию) регистрация активирует аккаунт сразу.
- После успешного логина клиент переводится в `/app`.
- После `POST /app/projects` сервер редиректит на `/app/p/{id}`, затем `/app/p/{id}/page.tsx` направляет на `/app/p/{id}/excel`.

## Source of truth and change policy

- Единственный источник истины: код в репозитории `schedule_builder_next`, который коммитится и пушится на GitHub.
- Все исправления делаются только через изменения файлов проекта (код, конфиги, docs) с последующей сборкой/деплоем.
- Прямые "исправления в браузере" не являются поддерживаемым способом разработки и не используются.
- Если наблюдаемое поведение UI расходится с ожидаемым, сначала перепроверяем собственную диагностику и фактический код/layout-контракт в репозитории.
- Любое финальное решение должно быть зафиксировано в GitHub-версии и отражено в документации.

## Export views contract

- `…/export/print` и `…/export/tilda` принимают `view` для области экспорта:
  - `timeline`, `tech-schedule`, `rooms`, `responsibles`, `vks`, `broadcasts`, `interpretation`, `volunteers`.
- Для `view=tech-schedule` берутся `tech_timeline_*`; для остальных областей — `timeline_*`.
- Ссылки на экспортные маршруты должны сохранять `view`, иначе пользователь попадает в дефолтную область.
